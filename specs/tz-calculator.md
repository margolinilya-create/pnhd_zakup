# ТЗ: Калькулятор закупщика швейного производства

**Версия документа:** 1.0
**Статус:** к разработке
**Назначение:** расчёт потребности в ткани и объёма закупки по заказу на пошив партии изделий, с фиксацией факта и самообучением на отклонениях план/факт.

---

## 1. Цель и границы продукта

### 1.1. Цель
Дать закупщику инструмент, который по модели изделия, ткани и размерному ряду рассчитывает:
- сколько ткани нужно на партию (в метрах и кг),
- сколько закупить с учётом резерва и кратности рулона,
- сколько это стоит в рублях,
и затем сравнивает план с фактом, накапливая статистику для уточнения нормативов.

### 1.2. В границах v1
- Справочники (ткани, поставщики, связи, изделия/паспорта).
- Расчёт потребности по партии (покомпонентно, по размерам, по нескольким тканям).
- Резерв и закупка с округлением до кратности рулона.
- Режимы «тест» и «заказ».
- Ввод факта и расчёт отклонения.
- Аналитика и рекомендации (полная спецификация, см. раздел 9).
- Роли и доступы.

### 1.3. Вне границ v1
- Вычитание складских остатков из закупки (валовая потребность; задел под v2 в модели данных есть).
- Автоматическое изменение коэффициентов системой (только рекомендации + ручное утверждение).
- Интеграция с конкретной внешней системой (Bitrix24 и т.п.) — заложена архитектурно, реализуется позже.

## 2. Глоссарий

- **SKU / модель** — конкретное изделие (например, «худи oversize»).
- **Паспорт изделия (спецификация, BOM)** — состав изделия: список компонентов, каждый со своей тканью, ролью и нормативом расхода.
- **Компонент** — часть изделия из одной ткани и роли (основное полотно, рибана, отделка).
- **Норматив расхода** — сколько ткани нужно на одно изделие базового размера для данного компонента, в канонической единице этой ткани.
- **Каноническая единица** — кг или метры; задаётся на уровне ткани; вторая единица всегда вычисляется.
- **Размерный коэффициент** — множитель расхода относительно базового размера (S=0.92, M=1.00, L=1.08 и т.д.).
- **Коэффициент раскладки** — небольшой множитель за перекос размерного состава и (для тканей в метрах) за ширину полотна.
- **Потери** — раскрой + пошив; усадка учитывается через предусадку ткани.
- **Резерв** — страховой запас в % от потребности.
- **Кратность рулона** — минимальный шаг закупки ткани у поставщика.
- **План / Факт / Отклонение** — расчётный расход, реальный расход, их разница.
- **Набор коэффициентов (coefficient set)** — версионируемый набор всех нормативов и коэффициентов на момент расчёта.

## 3. Роли и доступы

| Роль | Что может |
|---|---|
| **Технолог** | Ведёт справочники тканей/поставщиков, паспорта изделий, нормативы, размерные коэффициенты, потери. Создаёт новые версии набора коэффициентов. |
| **Закупщик** | Создаёт расчёты (тест), создаёт заказы, выбирает ткань/поставщика, размерный ряд, % резерва, курс. Видит результат и закупку. |
| **Оператор производства** | Вводит факт по закрытому заказу: реальный расход по ткани, отходы и брак с категорией. |
| **Аналитик / владелец** | Видит аналитику и отклонения, утверждает или отклоняет рекомендации по изменению коэффициентов. Доступ ко всему на чтение. |

Доступы реализуются на уровне БД (row-level security), а не только на фронте.

## 4. Архитектура

Пять логических слоёв. **Ключевой принцип: движок расчёта — чистая функция, не зависящая от платформы, хранилища и UI.**

```
[ UI / клиент ]
       │
[ Storage adapter ]  ←—— интерфейс репозитория (фронт не знает, где данные)
       │
[ ENGINE ]  ←—— чистая функция: (снимок справочников + спецификация заказа) → разложение расхода
       │
[ DATA ]   Master Data · Passports · Transactions · Coefficient sets · Analytics
```

- **Movok (Engine)** — детерминированный, без обращений к БД/сети. На вход получает уже собранный снимок данных, на выход отдаёт разложение. Покрыт unit-тестами.
- **Storage adapter** — интерфейс (`getFabric`, `getPassport`, `saveOrder`, …). Реализаций может быть несколько (Postgres/Supabase сейчас, Bitrix24 REST позже). UI и движок про конкретную реализацию не знают.
- **Иммутабельность транзакций** — заказ хранит снимок входов и версию набора коэффициентов; при изменении справочников история не пересчитывается.

## 6. Логика движка — порядок расчёта

Движок принимает снимок данных и спецификацию заказа, возвращает разложение. Шаги строго в этом порядке (важно для детерминизма):

**Шаг 1. Расход по компоненту и размеру (без потерь)**
Для каждого компонента `c` и каждого размера `s`:
```
raw(c) = Σ_s [ norm_base(c) × size_coef(s) × qty(s) ]
```
Результат — в канонической единице ткани компонента.

**Шаг 2. Множители**
```
net(c) = raw(c) × layout_coef(c) × (1 + loss_cut(c) + loss_sew(c)) × (1 + pre_shrink(fabric))
```
- `layout_coef(c)` — перекос размерного состава + (только для ткани с каноном «метры») влияние ширины относительно эталонной. Для ткани с каноном «кг» ширина в этот множитель почти не входит (вес не зависит от ширины при той же площади детали; ширина влияет лишь на малый процент отходов укладки).

**Шаг 3. Группировка по ткани**
Если одна ткань используется в нескольких компонентах/ролях — суммируем:
```
need(fabric) = Σ_{c: ткань(c)=fabric} net(c)
```

**Шаг 4. (v2, выключено в v1) Вычет складского остатка**
```
need_to_buy = need - stock   // в v1: need_to_buy = need
```

**Шаг 5. Резерв**
```
reserve(fabric)   = need_to_buy × reserve_pct
order_qty_raw     = need_to_buy × (1 + reserve_pct)
```

**Шаг 6. Округление до кратности рулона**
```
order_qty = ceil(order_qty_raw / roll_size) × roll_size   // в единице кратности
```

**Шаг 7. Стоимость**
```
cost = order_qty(в price_unit) × price
if currency == $:  cost_rub = cost × fx_rate
```

### Конверсия единиц (раздел 8) применяется там, где нужно показать вторую единицу или привести к единице цены/кратности.

## 7. Резерв, конверсия, вывод

### 7.1. Резерв
- Вводится вручную закупщиком в % (0 / 3 / 5 / 7 / 10 / своё).
- В v2 — рекомендация процента из аналитики (см. 9.4).

### 7.2. Конверсия метры ↔ кг
Масса погонного метра:
```
кг_на_пог.метр = ширина[м] × плотность[g/m²] / 1000
```
- канон «кг» → метры = кг / кг_на_пог.метр
- канон «м» → кг = м × кг_на_пог.метр

### 7.3. Валюта
- Цена хранится в валюте ребра поставщик×ткань (₽ или $).
- Курс — параметр заказа (`fx_rate`), фиксируется в транзакции.

### 7.4. Что выдаёт движок (result)
По каждой ткани:
- потребность (кг и м),
- резерв (кг и м),
- объём закупки с округлением до рулона,
- стоимость (₽),
И разложение факторов: вклад размеров, вклад потерь (раскрой/пошив отдельно), вклад ширины/раскладки, предусадка. Плюс общие итоги по заказу.

## 8. Режимы работы и факт

### 8.1. Режимы
- **Тест** — расчёт без сохранения как заказ; можно проверять сценарии.
- **Заказ** — сохраняется транзакция со снимком входов и версией коэффициентов; доступен для последующего ввода факта.

### 8.2. Факт (после производства)
Оператор производства по закрытому заказу вводит по каждой ткани: реальный расход, отдельно отходы и брак с категорией (брак ткани / брак пошива / естественные отходы), количество годных изделий.

### 8.3. Отклонение
```
deviation(fabric) = factual / plan - 1
```
Считается по ткани. Накапливается для аналитики.

## 9. Аналитика и самообучение (полная спецификация)

Система **советует, но не меняет коэффициенты сама**. Любое изменение проходит через утверждение аналитиком и создаёт новую версию набора коэффициентов.

### 9.1. Накопление
Хранит по каждому закрытому заказу: SKU, использованные ткани, размерную сетку, план, факт, отклонение, версию коэффициентов.

### 9.2. Атрибуция отклонения
Отклонение режется в разрезах, чтобы локализовать причину, а не давать общий «+8%»:
- по **SKU** (систематическая ошибка норматива модели),
- по **ткани** (плохая раскладка / нестабильная плотность),
- по **размерной сетке** (ошибка размерных коэффициентов),
- по **категории потерь** (раскрой vs пошив, из факта брака).

Рекомендация указывает конкретный коэффициент-кандидат и предлагаемую величину правки.

### 9.3. Уровень уверенности
Формально, а не «на глаз»:
```
confidence = f(n, σ)   // растёт с числом заказов n, падает с дисперсией отклонений σ
```
- Рекомендация выдаётся только при `n ≥ n_min` (например 5) и стабильном знаке отклонения.
- В рекомендации показывается: уверенность (%), n заказов, на которых она основана.

### 9.4. Аналитика по резерву
Отдельно отслеживает: хватало ли резерва, средний остаток после заказа, частоту нехватки по ткани → рекомендует оптимальный % резерва на ткань (вместо постоянного ручного).

### 9.5. Применение рекомендации
Аналитик утверждает → формируется новый CoefficientSet (version++). Старые заказы остаются на своей версии и не пересчитываются.

## 10. Интерфейс

### 10.1. Экран расчёта (закупщик)
Поля ввода: модель (SKU) → подтягивает компоненты паспорта; по каждому компоненту — выбор ткани (только из allowed_fabrics) и поставщика; размерный ряд и количество по размерам; курс $; % резерва.
Кнопки: **Рассчитать**, **Очистить**, **Сохранить как тест**, **Создать заказ**.
Блок результата (по каждой ткани и итог): расход (м и кг), резерв, итог к закупке (с округлением до рулона), стоимость (₽), расшифровка коэффициентов.

### 10.2. Справочники (технолог)
CRUD по тканям, поставщикам, рёбрам, SKU, паспортам, размерным коэффициентам, потерям. Создание версий набора коэффициентов.

### 10.3. Факт (оператор производства)
Список закрытых заказов → форма ввода факта по ткани + отходы/брак по категориям.

### 10.4. Аналитика (аналитик/владелец)
Таблица отклонений с разрезами; список рекомендаций с уверенностью и n; кнопки «утвердить/отклонить»; раздел аналитики резерва.

## 11. Технологический стек (рекомендация)

Архитектура — независимое ядро + адаптеры, поэтому стек выбираем под удобство, не запираясь.

- **Ядро (Engine):** TypeScript, чистые функции, ноль зависимостей от IO. Оформляется отдельным модулем/пакетом. Покрытие unit-тестами (Vitest), включая золотые кейсы расчёта. Этот модуль переносится без изменений в любой будущий хост, включая Bitrix24.
- **UI:** React + TypeScript (Vite). По духу близко к уже знакомому формату одностраничного приложения, но с реальной типизацией и структурой.
- **Хранилище + auth + роли:** Supabase (Postgres). Закрывает БД, аутентификацию и **row-level security** под четыре роли одним сервисом — это снимает отдельную разработку бэкенда на v1.
- **Storage adapter:** интерфейс репозитория между UI/движком и Supabase. Когда понадобится Bitrix24 — пишется вторая реализация адаптера (маппинг сущностей на смарт-процессы через REST), ядро и UI не трогаются.

Почему так: движок изолирован и тестируем; persistence/роли берутся «из коробки»; путь на Bitrix24 открыт без переписывания.

## 12. Нефункциональные требования

- **Детерминизм:** одинаковый вход + одна версия коэффициентов = одинаковый результат до копейки/грамма. Порядок округлений зафиксирован (раздел 6).
- **Иммутабельность истории:** заказы хранят снимок входов и версию коэффициентов.
- **Тестируемость:** движок покрыт золотыми кейсами; правки коэффициентов не ломают исторические расчёты.
- **Точность:** хранить расчётные значения без преждевременного округления; округлять только на отображении и на шаге кратности рулона.

---

## 13. Этапы разработки

1. **Ядро + модель данных.** Сущности, движок (шаги 1–7 без склада), unit-тесты, конверсия единиц/валют.
2. **Справочники и паспорта (BOM).** CRUD, версии коэффициентов, роли.
3. **Расчёт и заказы.** Экран расчёта, режимы тест/заказ, снимок и иммутабельность.
4. **Факт и отклонение.** Ввод факта, отходы/брак, расчёт отклонения по ткани.
5. **Аналитика.** Накопление, атрибуция, уверенность, рекомендации, аналитика резерва.
6. **(v2)** Складские остатки в расчёте закупки; адаптер под Bitrix24.

## 14. Открытые вопросы / допущения

- Размерные коэффициенты на старте задаются seed-таблицей по категориям и уточняются аналитикой; точные значения требуют замеров на нескольких моделях.
- Предполагается, что закупщик выбирает поставщика вручную; авто-выбор «дешевле/быстрее» — кандидат на v2.
- Кратность рулона берётся с ребра поставщик×ткань; если у поставщика не задана — fallback на Fabric.roll_size.
