# PROJECT CONTEXT: Калькулятор закупщика швейного производства

> Этот файл — полное саммори проектной сессии. Положи в корень репозитория как `PROJECT_CONTEXT.md` и ссылайся на него из `CLAUDE.md`. Источник истины по домену — `specs/tz-calculator.md`; этот файл объясняет, ЗАЧЕМ всё устроено именно так и какие решения уже приняты.

## 1. Что строим

Внутренний инструмент для закупщика швейного производства. По модели изделия (SKU), размерному ряду, ткани и поставщику считает: потребность в ткани (метры и кг), резерв, объём закупки с округлением до рулона, стоимость в ₽. Сохраняет расчёт как заказ, потом сравнивает план с фактом и (в v2) учится на отклонениях.

Бизнес-контекст: реальное швейное производство (РФ, СПб), два поставщика тканей — МЕДАС и КОТОНПРОМ. Данные взяты из двух Google Sheets пользователя: «База тканей» и «База SKU / PRODUCT_PASSPORT».

## 2. Целевой стек и фундамент

- Фундамент: шаблон **vibe** (https://github.com/di-sukharev/vibe), залит в репозиторий пользователя.
- Backend: **Bun + Hono + Prisma + PostgreSQL**, JWT-auth шаблона, Zod-валидация.
- Web: **React + Vite + TanStack** (Query/Form/Router).
- Контракты: **packages/contracts** — общие Zod-схемы + чистый движок расчёта.
- **База данных: Supabase** (managed PostgreSQL 17, проект `pnhd-zakup`, регион eu-central-1 / Франкфурт). Подключение приложения и с локальной машины — только через **Session Pooler** (IPv4; прямой хост `db.*.supabase.co` — IPv6-only и недоступен). Локального Docker-Postgres больше нет. Тесты идут в отдельной схеме **`app_test`** той же базы. Секреты/строки подключения — в `backend/.env` (в `.gitignore`). Детали — `docs/LOCAL_DATABASE.md`.
- **Деплой:** фронтенд (`webapp`) **и** backend (Hono) — оба на **Vercel**, автодеплой с ветки `master`:
  - webapp → https://pnhd-zakup-webapp.vercel.app (проект `pnhd-zakup-webapp`, root `webapp`); сборочный `VITE_API_URL=https://pnhd-zakup-api.vercel.app`.
  - API → https://pnhd-zakup-api.vercel.app (проект `pnhd-zakup-api`, root `backend`); хэндлер — самописный буферизующий мост `req→Hono` в `src/server-entry.ts` (стандартный node-server streaming-адаптер виснет на non-GET в раннере Vercel). Env: `DATABASE_URL` (session pooler), `JWT_SECRET`, `CORS_ORIGINS`=origin webapp, `COOKIE_SECURE=true`.
  - Миграции БД накатываются вручную: `bun run --cwd backend prisma:deploy` (читает `DATABASE_URL` прод-схемы `public` из `backend/.env`). Vercel-сборка миграции не запускает. Детали — `docs/DEPLOYMENT.md`.
- **Роль Supabase = только managed Postgres.** Подключаемся ролью `postgres` через Prisma; **RLS выключен**; Supabase Auth/Storage не используем. Аутентификация — JWT шаблона, авторизация по ролям — на backend (RBAC в v2). Рекомендация ТЗ §11 (Supabase Auth + RLS, без отдельного бэкенда) **сознательно не взята** в пользу пути шаблона vibe.
- Активные поверхности: **web + backend/API**. mobile и landing — отложены (deferred-заметки в их README).
- Workflow разработки: **bulletproof** (фазы → TDD → гейты → коммит → свежий контекст). Гейты: `bun run typecheck` · `bun run test` · `bun run e2e:web`.

> **Инфраструктура изменена относительно исходного плана.** Первоначально планировались локальный Docker-Postgres + продакшн на DigitalOcean/Yandex Cloud — этот путь **отменён** в пользу Supabase + Vercel; гайды в `docs/` приведены в соответствие. Важно для РФ-производства: Supabase размещён в Германии — если встанет вопрос **152-ФЗ** (хранение перс. данных в РФ), пересмотреть хостинг (Yandex Cloud остаётся задокументированной альтернативой в `docs/YANDEX_CLOUD.md`).

## 3. Ключевые архитектурные решения (зафиксированы, не пересматривать без причины)

1. **Движок расчёта — чистая функция** в `packages/contracts`, без Prisma/сети/env. Сервис собирает снимок справочников из БД и передаёт в движок. Причина: платформа может смениться (позже — интеграция с Bitrix24 через адаптер), ядро должно переноситься без переписывания.
2. **Паспорт изделия = BOM (спецификация)**: SKU → список компонентов (роли MAIN/RIB/TRIM/OTHER), у каждого своя ткань, норматив, потери, список допустимых тканей. В v1 у всех паспортов один компонент MAIN (в исходных данных рибана/манжеты не выделены), но модель уже многокомпонентная.
3. **Размерные коэффициенты — таблица** {XS:0.85, S:0.92, M:1.0, L:1.09, XL:1.18, XXL:1.28}, а не один общий «коэффициент перекоса». Базовый размер — M (соответствует AVG в исходной базе).
4. **Цена — на ребре «поставщик × ткань»** (SupplierFabric), не на ткани. Одна ткань у двух поставщиков по разной цене.
5. **Каноническая единица — на уровне ткани**. В данных пользователя весь трикотаж в кг → канон 'kg'; метры выводятся через массу пог. метра = ширина(м) × плотность(g/m²) / 1000.
6. **Заказы иммутабельны**: Order хранит `input_snapshot` (снимок входов и использованных записей справочников) + `result`. Изменение справочника ПОСЛЕ заказа не меняет сохранённый расчёт. Это главный инвариант проекта — нужен для честного сравнения план/факт.
7. **Версионирование коэффициентов** (CoefficientSet) и аналитика с атрибуцией отклонений — спроектированы в ТЗ, реализуются в v2.
8. **Склад в v1 не вычитается** — считаем валовую потребность. Задел под вычет остатков (v2) в модели есть.
9. **Роли**: User.role enum TECHNOLOGIST / PURCHASER / OPERATOR / ANALYST (дефолт PURCHASER). В v1 — только поле; полноценный RBAC в v2.
10. **Факт вводится вручную**, отходы/брак — отдельно по категориям (брак ткани / брак пошива / естественные отходы). Отклонение = факт/план − 1, считается по ткани.

## 4. Формула движка (раздел 6 ТЗ, реализована в Фазе 1)

Порядок строгий (для детерминизма):
- шаг 1: по компоненту `raw = norm_base × Σ(size_coef[s] × qty[s])`
- шаг 2: `net = raw × layout_coef × (1 + loss_cut + loss_sew) × (1 + pre_shrink)`
  - `layout_coef = perekos × widthEffect`, REF_WIDTH = 180
  - perekos: если макс. доля одного размера > 0.5 → `1 + (доля − 0.5) × 0.1`, иначе 1
  - widthEffect: канон 'm' → `REF_WIDTH/width`; канон 'kg' → `1 + max(0,(REF_WIDTH−width)/REF_WIDTH) × 0.05`
- шаг 3: группировка по ткани (одна ткань в нескольких ролях — суммируется)
- шаг 5: `reserve = need × reserve_pct`
- шаг 6: `order = ceil(toBuy / roll_size) × roll_size` в единице рулона
- шаг 7: `cost = order(в price_unit) × price`; USD → `× fx_rate`
- Конверсия кг↔м: `кг_на_м = width_m × density_gsm / 1000`. Без промежуточных округлений; округлять только на шаге рулона.
